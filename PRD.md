# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 배경

가상의 반도체 회사 **S-Semi**는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등에 납품한다.

- 시료는 주문 접수 → 웨이퍼 공정 설비를 통한 제작 → 검수 → 고객 출고 순으로 처리된다.
- 최근 주문량 급증으로 다음과 같은 문제가 발생했다.
  - 주문 처리 여부를 파악하기 어려움
  - 공정 예약 후 완성 시점을 알 수 없음
  - 재고가 충분함에도 불필요한 추가 공정이 진행됨
  - 엑셀/메모장 기반 관리로 인한 잦은 실수, 재고·공정 현황 파악 불가

이를 해결하기 위해 **"반도체 시료 생산주문관리 시스템"**을 개발한다.

## 2. 목적

주문 접수부터 생산, 출고까지의 전 과정을 체계적으로 관리하고, 담당자가 재고·주문·생산 현황을 한눈에 파악할 수 있는 콘솔 기반 관리 시스템을 구축한다.

## 3. 역할 및 흐름

| 역할 | 설명 |
|---|---|
| 고객 | 필요한 시료를 이메일 등으로 요청 (시스템 외부 행위) |
| 주문 담당자 | 고객 요청에 맞게 주문서(주문) 작성 및 관리 |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 생산 관리 |

**처리 흐름**: 고객 → (시료 요청) → 주문 담당자 → (주문서 전달) → 생산 담당자 → (승인/거절) → 주문 담당자

## 4. 시스템 개요

- **생산 라인**: 공장에서 시료 하나를 생산하는 설비 흐름. 하나의 생산 라인은 시료를 하나씩 생산하며, 주문이 들어온 시료에 대해서만 생산한다.
  - **시스템 전체에 생산 라인은 1개(단일 라인)이다.** 시료 종류와 무관하게 모든 생산 대기 작업은 하나의 FIFO 큐에서 순차적으로 처리된다. (병렬 생산 없음)
- **동작 방식**: 콘솔 기반. 담당자가 직접 명령을 입력하여 시료 등록 및 주문 처리를 수행한다.

## 5. 주문 상태 흐름

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 외 상태, 모니터링 대상에서 제외) |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 |
| RELEASE(D) | 출고 완료 |

**상태 전이도**:
```
주문 등록 → RESERVED
  → (승인 여부: 거절) → REJECTED
  → (승인 여부: 승인) → 재고 확인
      → (재고 충분) → CONFIRMED → (출고 처리) → RELEASE
      → (재고 부족) → PRODUCING(생산 요청) → (생산 완료) → CONFIRMED → (출고 처리) → RELEASE
```

## 6. 기능 요구사항

### 6.1 메인 메뉴
- 기능별 선택 화면 표시
- 전체 시료에 대한 요약 정보 표시(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수 등)
- 하위 메뉴: 시료 관리 / 시료 주문 / 주문 승인·거절 / 모니터링 / 생산라인 조회 / 출고 처리

### 6.2 시료 관리
시료(Sample)는 시스템의 가장 기본이 되는 단위이며, 등록된 시료만 주문 가능하다.

- **시료 등록**: 새로운 시료를 시스템에 추가
  - 속성: 시료 ID, 이름, 평균 생산시간, 수율
  - 수율 = 정상적인 시료 수 / 총 생산 시료 수 (예: 100개 중 90개 정상 → 0.9)
  - **시료 ID는 시스템이 자동 채번한다.** 형식: `S-XXX` (예: `S-001`, `S-002`, ... 3자리 순번, 등록 순서대로 증가)
- **시료 조회**: 등록된 모든 시료 목록과 현재 재고 수량 표시
- **시료 검색**: 이름 등 속성으로 특정 시료 검색

**추가 제약사항**
- 시료 재고는 프로그램을 종료했다가 다시 실행하더라도 계속 유지·반영되어야 한다. (재고 데이터는 영속 저장소에 기록)

### 6.3 시료 주문 (예약)
고객이 시료를 요청하면 주문 담당자가 주문을 생성한다.

- 입력 값: 시료 ID, 고객명, 주문 수량
- 생성 시 주문 상태는 `RESERVED`
- **주문번호는 시스템이 자동 채번한다.** 형식: `ORD-YYYYMMDD-NNNN` (예: `ORD-20260416-0043`, 등록일자 + 해당 일자 내 4자리 순번)

### 6.4 주문 승인/거절
- **접수된 주문 목록**: `RESERVED` 상태 주문 목록 표시
- **주문 승인**: 재고 상황에 따라 자동 분기 처리
  - 재고 충분 → 즉시 `CONFIRMED`로 전환
  - 재고 부족 → 생산 라인에 자동 등록, 상태 `PRODUCING`으로 전환
- **주문 거절**: 즉시 `REJECTED`로 전환

**추가 제약사항**
- 재고 판단은 **주문을 승인하는 시점**의 재고 현황을 기준으로 한다. (예약(RESERVED) 시점이 아닌, 승인 처리 시점의 실시간 재고로 충분/부족 여부를 판단)
- 이미 승인되어 재고에서 선점(차감)된 수량은, 이후 처리되는 다른 주문에서 **중복으로 사용할 수 없다.** 즉, 승인 처리 시 내부적으로는 재고를 즉시 차감(또는 잠금)하여 뒤이은 주문의 재고 확인(충분/부족 판단)에 반영되도록 한다.
- 단, 위와 같이 선점되어 내부적으로 차감된 재고 수량은 **주문이 종료(출고 완료, RELEASE)되기 전까지는 사용자에게 보여지는 재고 수량(시료 조회, 모니터링 등)에서 차감된 것으로 표시하지 않는다.** 즉, 재고는 두 가지로 구분하여 관리한다.
  - **내부 판단용 가용 재고**: 승인 시점에 즉시 차감되어, 다음 주문의 승인 가능 여부 판단에 사용됨 (중복 선점 방지)
  - **화면 표시용 재고**: 실제 물리적 재고 수량을 그대로 표시하며, 선점한 주문이 RELEASE로 종료되어야 비로소 차감된 수치가 반영됨
- **내부 판단용 가용 재고의 차감 범위**: 재고가 부족해 `PRODUCING`으로 전환되는 주문의 경우, 내부 가용 재고는 **물리 재고만큼만 차감하고 0에서 멈춘다** (마이너스로 내려가지 않음). 부족분(생산이 필요한 수량)은 생산 완료 시점에 화면 표시용 재고에 반영되며, 그 전까지는 내부 가용 재고 계산에 별도로 포함되지 않는다.

### 6.5 모니터링
- **주문량 확인**: 상태별(RESERVED/CONFIRMED/PRODUCING/RELEASE) 주문 건수 표시 (REJECTED는 무효 주문이므로 제외)
- **재고량 확인**: 시료별 현재 재고 수량 및 상태 표시
  - 여유: 주문 대비 재고 충분
  - 부족: 주문 대비 재고 수량 부족
  - 고갈: 재고 수량 0
  - 여기서 표시되는 재고 수량은 **화면 표시용 재고**([6.4 주문 승인/거절] 참고)이며, 승인 시 선점(내부 차감)된 수량은 해당 주문이 RELEASE로 종료되기 전까지 차감 반영하지 않는다.

### 6.6 생산라인
- 주문량 대비 부족분을 수율 및 오차를 고려하여 생산
  - 실 생산량 = `ceil(부족분 / 수율)`
  - 총 생산 시간 = `평균 생산시간 * 실 생산량`
  - 생산 완료 시 주문 상태 `PRODUCING → CONFIRMED` 전환
- **생산 현황 표기**: 현재 생산 중인 시료 정보 표시(주문 정보, 현재까지 생산량 등, 표기 수준은 자율)
- **대기 주문 확인**: 생산 큐(대기열)에 있는 대기 작업 목록 표시
  - 스케줄링 전략: **FIFO** (선입선출)
  - 생산 라인은 시스템 전체에 1개뿐이므로, 큐에 쌓인 모든 대기 작업(시료 종류 무관)은 이 순서대로 하나씩 순차 처리된다.

**추가 제약사항**
- 생산 소요 시간은 **현실 시간(real time)**에 맞춰 진행되어야 한다. (예: 총 생산 시간이 10분이면 실제로 10분이 경과해야 완료)
- **시간 단위**: `평균 생산시간` 속성의 단위는 **분(min/ea)**이며, PDF 예시와 동일하게 **소수점 입력을 허용**한다 (예: `0.1`min = 6초). (과거 결정이었던 "ms 단위로 정의" 방침을 대체함 — 데모에서 현실적인 대기 시간이 필요하면 등록 시 작은 소수 값을 입력해 조절한다.) 내부적으로 실시간 타이머(Phase 9)를 계산할 때는 이 값을 밀리초로 환산(`분 * 60000`)해 사용한다.
- 생산이 **완료된 시점에만** 재고에 반영한다. 생산 중(PRODUCING)인 수량은 재고에 포함되지 않는다.
- 프로그램이 생산 도중 종료되었다가, 생산 완료 시점 이후에 다시 실행되었을 경우에도 생산이 완료된 상태(및 그에 따른 재고 반영, 주문 상태 전환)로 복원되어야 한다. (생산 시작 시각/예상 완료 시각을 영속 데이터로 저장하고, 재기동 시 현재 시각과 비교하여 완료 여부를 판단)
- 수율을 반영해 실 생산량을 늘려서 생산하지만(`ceil(부족분 / 수율)`), 생산된 수량은 **전량 정상 판정**되어 재고에 그대로 반영한다. (불량품으로 인한 손실 개념은 시뮬레이션하지 않음)
- 생산 큐에 한 번 들어간 주문은 **무조건 생산이 진행되어야 한다.** (대기 중 임의 취소·재고 유입 등에 의한 생산 스킵 없음)

### 6.7 출고 처리
- `CONFIRMED` 상태 주문에 대해 출고 실행
- 처리 완료 시 주문 상태 `CONFIRMED → RELEASE`로 전환

## 7. 비기능 요구사항 / 개발 지침

### 7.1 미션 1 — PoC 개발
프로젝트 구현 전 개별 Repository로 아래 PoC를 각각 검증한다.

| PoC 항목 | 설명 |
|---|---|
| MVC 스켈레톤 코드 | Model / Controller / View 패키지 구조와 역할 분리 완성 |
| 데이터 영속성 처리 | **JSON 파일 기반**(nlohmann/json 라이브러리)으로 데이터 저장·조회 구조 구현, CRUD 포함. (애플리케이션 재실행 후에도 데이터 유지) |
| 데이터 모니터링 Tool | 저장된 데이터 상태를 콘솔에서 실시간 조회 가능한 관리자 도구. 특정 조건(예: 이름, ID 등 속성)으로 데이터를 검색하는 기능 포함 |
| Dummy 데이터 생성 Tool | 테스트용 Dummy Data 생성 도구 (연결된 DB에 추가) |

**영속성 기술 결정**: 데이터 영속성은 **JSON 파일 기반**(nlohmann/json, NuGet 패키지 `nlohmann.json`)으로 구현한다. PoC(`DataPersistence`)와 메인 프로젝트(`SampleOrderSystem`)는 코드를 그대로 재사용하지 않더라도, **동일한 라이브러리와 동일한 Repository 인터페이스 사용법**(예: `save`/`load`/`findById`/`findAll`)을 따른다.

### 7.2 미션 2 — 프로젝트 개발
Agentic Engineering을 도입하여 기능 명세를 충족하는 고품질 코드를 개발한다. 주안점:

1. CLAUDE.md, PRD.md 등 문서 관리
2. Harness 도입
3. Test
4. Clean Code
5. Commit 이력 관리

### 7.3 테스트 정책
- **gmock(Google Mock)**을 이용한 테스트를 개발 단계마다(기능 구현 시마다) 작성한다.
- 외부 의존성(데이터 저장소, 시간/타이머, 생산라인 등)은 Mock 객체로 대체하여 단위 테스트를 수행한다.
- 신규 기능 추가 및 기존 기능 수정 시, 관련 테스트 코드를 함께 작성·갱신한다.

## 9. UI 참고사항

PDF에 수록된 콘솔 화면(메인 메뉴, 시료 관리, 시료 주문, 주문 승인/거절, 모니터링, 생산라인, 출고 처리)은 **이해를 돕기 위한 예시**이며, 실제 화면 구성은 자유롭게 결정 가능하다.

## 10. 과제 진행 유의사항

- 지정된 모델(Sonnet / Effort: Medium)만 사용
- Model 사용 이력이 없거나 Opus 사용 이력이 있으면 부정행위로 간주, 동일 조건으로 평가
- 과제 종료 시 Claude Code에서 `/logout` 실행 필요
