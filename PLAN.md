# PLAN: 반도체 시료 생산주문관리 시스템 개발 계획

이 문서는 `PRD.md`의 요구사항을 구현 가능한 단위(Phase)로 분해한 실행 계획이다.

각 Phase는 별도 세션에서도 이어서 진행할 수 있도록 목표 / 작업 항목 / 산출물 / 완료 기준(Exit Criteria)을 명시한다.

## Phase 0 — 저장소 초기 셋업

**목표**: PoC 4종 + 메인 프로젝트, 총 5개의 독립 저장소를 로컬/원격에 준비한다.

**작업 항목**
1. `Semiconductor/` 하위에 저장소별 폴더 생성 (각 폴더는 독립된 git 저장소)
   - `ConsoleMVC-JeonHyunji-10225419/`
   - `DataPersistence-JeonHyunji-10225419/`
   - `DataMonitor-JeonHyunji-10225419/`
   - `DummyDataGenerator-JeonHyunji-10225419/`
   - `SampleOrderSystem-JeonHyunji-10225419/`
2. 각 폴더에 대해:
   - `git init`
   - 기본 구조 생성: `README.md`, `.gitignore`(Visual Studio/C++용), `CLAUDE.md`(저장소별 개발 가이드)
   - 초기 커밋 (`chore: initial commit`)
3. GitHub CLI(`gh`)로 각 저장소를 **Public**으로 생성 후 최초 push
   - `gh repo create {repo명} --public --source=. --remote=origin --push`
4. 5개 저장소 모두 원격 push 완료 확인 (`git remote -v`, `gh repo view`)

**산출물**: 5개 폴더 + 5개 GitHub Public 저장소, 각 초기 커밋 1개씩

**완료 기준**
- 사용자에게 본인 영문이름/사번 확인 후 폴더명 확정
- `gh auth status` 로그인 확인
- 5개 저장소 모두 GitHub 상에서 Public으로 조회 가능

**주의**: GitHub push는 외부에 반영되는 되돌리기 어려운 작업이므로, 각 저장소 생성/push 직전에 사용자 확인을 받는다.

## Phase 8 — 주문 승인/거절 기능 (재고 선점 로직 포함)

**목표**: PRD 6.4 요구사항 + 추가 제약사항(승인 시점 재고 판단, 재고 이중 관리) 구현

**작업 항목**
- `RESERVED` 목록 조회
- 승인 처리:
  - **승인 시점**의 내부 가용 재고로 충분/부족 분기
  - 재고 충분 → `CONFIRMED` 전환 + **내부 가용 재고를 주문 수량만큼 즉시 차감**(화면 표시 재고는 유지)
  - 재고 부족 → 생산 큐 등록 + `PRODUCING` 전환 + **내부 가용 재고는 남아있는 물리 재고만큼만 차감하여 0에서 멈춤** (마이너스로 내려가지 않음; 부족분은 생산 완료 후 화면 표시 재고에만 반영)
- 거절 처리: `REJECTED` 전환 (내부/화면 재고 모두 변화 없음)
- **재고 이중 관리 구조** 설계: `physicalStock`(화면 표시용, RELEASE 시점에만 감소) vs `availableStock`(내부 판단용, 승인 시점에 즉시 감소)

**완료 기준**
- 동일 시료에 대해 순차 주문 2건 승인 시, 두 번째 주문이 첫 번째가 선점한 재고를 중복 사용하지 않음을 테스트로 검증
- 승인 후 시료 조회(화면) 시 재고 수량이 아직 차감되지 않은 것으로 표시됨을 검증
- gmock 기반 승인 로직 단위 테스트 (재고 충분/부족/거절 각 케이스)

## Phase 9 — 생산라인 기능 (실시간 처리 + 재기동 복원)

**목표**: PRD 6.6 요구사항 + 추가 제약사항(현실시간 반영, 생산 완료 시 재고 반영, 재기동 복원, 수율 전량 정상 판정, FIFO 큐, 큐 진입 시 무조건 생산) 구현

**작업 항목**
- 생산 큐(FIFO) 자료구조 및 영속화 (대기 목록이 재시작 후에도 유지되어야 함). **생산 라인은 시스템 전체 1개(단일 라인)** — 시료 종류 무관하게 이 큐 하나로 순차 처리
- 실 생산량/총 생산시간 계산: `ceil(부족분/수율)`, `평균생산시간(min/ea, 소수 가능) * 실생산량` — 저장 단위는 **분(소수 허용, 예: 0.1)**, 실시간 타이머 계산 시 밀리초로 환산(`분 * 60000`)해서 사용
- **현실시간 타이머**: 생산 시작 시각 + 예상 완료 시각(ms 단위 계산)을 영속 저장 → 별도 스레드/타이머로 진행률 추적
- **재기동 복원 로직**: 프로그램 시작 시 저장된 생산 작업의 완료 시각과 현재 시각 비교
  - 이미 완료 시각이 지났다면 → 즉시 완료 처리(재고 반영 + `PRODUCING → CONFIRMED`)
  - 아직 진행 중이면 → 잔여 시간 재계산 후 타이머 재개
- 생산 완료 시: 화면 표시용 재고(`physicalStock`)에 실생산량 전량 반영 (수율 무관, 전량 정상 판정)
- 생산 현황 조회, 대기 큐 조회(FIFO 순서) 화면 구현

**완료 기준**
- 생산 중 프로그램 종료 → 완료 시각 이후 재실행 시 자동 완료 처리 검증 (테스트에서는 시간 주입/Mock으로 시뮬레이션)
- 생산 진행 중에는 재고 미반영, 완료 후에만 반영됨을 테스트로 검증
- gmock으로 시간 소스(Clock)를 목킹하여 실시간 의존성 없는 단위 테스트 작성

## 진행 방식 메모

- Phase 0은 되돌리기 어려운 GitHub 관련 작업(저장소 생성/push)을 포함하므로, 실행 직전 매번 사용자 확인을 받는다.
- Phase 1~4(PoC)는 각자 독립 저장소이므로 병렬로 진행 가능하나, 기본은 순서대로 진행한다.
- Phase 5 이후 메인 프로젝트는 PoC의 "코드 이식"이지 "저장소 연결(submodule)"이 아님에 유의한다.
- Phase 8, 9는 재고 이중 관리·실시간 처리라는 프로젝트의 핵심 난이도 구간이므로, 각 Phase 완료 후 반드시 완료 기준의 테스트 시나리오를 실행해 검증한다.
